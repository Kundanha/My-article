HTTP Protocol - Complete Guide

üì° HTTP Kya Hai? (What is HTTP?)

HTTP (Hypertext Transfer Protocol) ek application-level protocol hai jo World Wide Web ke foundation par kaam karta hai. Ye 1989 mein Tim Berners-Lee ne invent kiya tha.

üéØ Basic Working

  • Client-Server Model: Browser (client) request bhejta hai ‚Üí Server response deta hai
  • Web Browser = User Agent (UA) - jo request bhejta hai
  • Web Server = Jo websites host karta hai aur response deta hai
  • Stateless Protocol: Har request independent hai, server previous requests yaad nahi rakhta

üìä HTTP Versions Ki Evolution

HTTP/0.9 (1991): Sirf GET method, sirf HTML response

HTTP/1.0 (1996): Headers add hue, multiple content types support

HTTP/1.1 (1997): Persistent connections, pipelining, chunked transfer

HTTP/2 (2015): Binary protocol, multiplexing, server push, header compression

HTTP/3 (2022): QUIC + UDP, faster connections, less latency

üí° Safe vs Idempotent Samjho:

üîí Safe Method: Server pe koi change nahi karta (read-only). Example: GET, HEAD

üîÑ Idempotent Method: Ek baar ya 100 baar call karo, same result. Example: GET, PUT, DELETE

POST idempotent nahi hai kyunki har call naya resource create karega!

‚ö° HTTP/2 & HTTP/3 Improvements

  • Multiplexing: Single TCP connection pe multiple requests/responses - no head-of-line blocking!
  • Header Compression (HPACK): Repeated headers compress hote hain - bandwidth save
  • Server Push: Server client ke maange bina resources bhej sakta hai
  • Binary Protocol: Text-based se binary - faster parsing
  • HTTP/3 uses QUIC: UDP-based, no TCP handshake delay, faster recovery from packet loss

üç™ Sessions & Cookies

HTTP stateless hai, toh sessions maintain karne ke liye:

  • HTTP Cookies: Server session ID cookie mein store karta hai
  • Hidden Form Fields: Session data forms mein hidden inputs mein
  • URL Rewriting: Session ID URL mein embed (less secure)

üîå Persistent Connections (Keep-Alive)

HTTP/1.0 Problem: Har request ke liye naya TCP connection - expensive!

HTTP/1.1 Solution: TCP connection reuse multiple requests ke liye

  • TCP 3-way handshake bar-bar nahi karna padta
  • Connection time ke saath faster hota hai (TCP slow-start)
  • Latency significantly reduced!

üì¶ Content Retrieval Optimizations

  • Conditional GET: If-Modified-Since header - sirf changed content download karo
  • ETag: Entity tag - check karo ki cached version same hai ya nahi
  • Chunked Transfer Encoding: Content stream karo chunks mein - dynamic content ke liye
  • Byte Range Serving: File ka specific portion request karo - resume downloads!
  • Content-Encoding (gzip): Compressed content - bandwidth save

üåê Web Caching & Proxies

HTTP intermediate network elements ko allow karta hai:

  • Web Cache Servers: Upstream servers ki taraf se content deliver karte hain - response time improve
  • Browser Cache: Previously accessed resources store karta hai - network traffic reduce
  • Proxy Servers: Private network boundaries pe - clients ke liye communication facilitate
  • CDN (Content Delivery Networks): Globally distributed caches

üîê HTTP Authentication

  • Basic Authentication: Username:password base64 encoded (not secure without HTTPS!)
  • Digest Authentication: Password hash bhejta hai, zyada secure
  • Bearer Token: OAuth tokens - modern APIs mein common
  • Realms: Different authentication scopes under one root URI

üéØ Interview Questions - HTTP Deep Dive

Q: HTTP stateless hai toh sessions kaise maintain hoti hain?

A: Cookies! Server Set-Cookie header bhejta hai, browser har subsequent request mein Cookie header bhejta hai. Session ID cookie mein store hota hai jo server-side session data se map hota hai. Alternatives: JWT tokens, URL parameters (legacy).

Q: PUT vs PATCH mein kya difference hai?

A: PUT = Complete replacement (poora resource bhejo). PATCH = Partial update (sirf changed fields bhejo). PUT idempotent hai, PATCH technically nahi but practically mostly idempotent implement hota hai.

Q: CORS kya hai aur kyun zaroori hai?

A: Cross-Origin Resource Sharing - browser security feature. Same-origin policy ke under ek domain doosre domain ke resources access nahi kar sakta. CORS headers (Access-Control-Allow-Origin) explicitly permission dete hain. Preflight OPTIONS request hota hai complex requests ke liye.

Q: HTTP/2 Server Push kab use karna chahiye?

A: Jab aapko pata ho ki client ko kaunse resources chahiye honge (CSS, JS files HTML ke saath). Lekin careful - over-pushing se bandwidth waste hoti hai. Chrome ne actually Server Push deprecate kar diya hai kyunki caching complexities ki wajah se. Better alternative: Resource Hints (preload, prefetch).

‚ö†Ô∏è Relational DB Ki Limits - Warning Signs

Agar ye symptoms dikh rahe hain toh shayad relational database outgrow ho raha hai:

üèóÔ∏è Structural Symptoms

  1. Sparse Tables: Bahut saare columns hain but mostly empty - wastage!
  2. Attribute Tables: (foreign_key, attribute_name, value) pattern use ho raha hai - complex joins!
  3. Serialized Data: JSON/XML ek column mein store kar rahe ho kyunki schema accommodate nahi kar paati
  4. Complex Relationships: Bahut saare many-to-many join tables, recursive foreign keys
  5. Frequent Schema Changes: Baar baar ALTER TABLE karna pad raha hai

üìà Scalability Symptoms

  1. Write Bottlenecks: Single database server ki write capacity hit ho rahi hai
  2. Data Volume: Data itna bada ho gaya ki single server pe fit nahi ho raha
  3. Performance Issues: Batch processes ya analytical queries transactional workload slow kar rahe hain

üî∫ CAP Theorem & PACELC

CAP Theorem (Brewer's Theorem)

Distributed system mein sirf 2 out of 3 guarantee ho sakti hai:

üîÑ
Consistency

Har read latest write return kare

‚úÖ
Availability

Har request ka response mile (success ya failure)

üîå
Partition Tolerance

Network partition hone pe bhi system chale

Real-World Trade-offs:

Choice Sacrifice Examples
CP Availability MongoDB, HBase, Redis
AP Consistency Cassandra, CouchDB, DynamoDB
CA Partition Tolerance Traditional RDBMS (single node)

Note: P (Partition Tolerance) practically zaroori hai distributed systems mein - network failures happen!

PACELC Extension:

Partition hone pe ‚Üí Availability ya Consistency choose karo

Else (no partition) ‚Üí Latency ya Consistency choose karo

Basically: Even when network is fine, there's a trade-off between speed and consistency!

üéØ Interview Questions - CAP Theorem

Q: "Banking system mein CAP ka kaunsa combination use karoge aur kyun?"

Answer: Banking mein CP (Consistency + Partition Tolerance) choose karenge. Kyunki:

  • Account balance mein inconsistency acceptable nahi hai - agar do ATMs se same account se paise nikle toh disaster!
  • Network partition hone pe, system temporarily unavailable ho sakta hai, lekin galat balance show nahi hona chahiye
  • Read-your-writes guarantee zaroori hai - deposit ke baad balance reflect hona chahiye

Example: MongoDB, HBase use karte hain CP mode mein

Q: "Social media feed ke liye kaunsa CAP combination best hai?"

Answer: Social media ke liye AP (Availability + Partition Tolerance) best hai:

  • Agar kuch seconds puraana feed dikhe - koi problem nahi (eventual consistency ok)
  • Lekin app down ho - users chale jayenge! (availability critical)
  • Like count mein 1-2 ka difference acceptable hai temporarily

Example: Cassandra (Facebook), DynamoDB (Amazon) use karte hain

Q: "CAP mein 'Partition Tolerance' practically ignore kyun nahi kar sakte?"

Answer: Network partition inevitable hai distributed systems mein:

  • Cables cut ho sakte hain, routers fail ho sakte hain
  • Cloud regions ke beech connectivity issues common hain
  • Even within datacenter, network glitches hote hain
  • Isliye real-world mein choice hai: CP vs AP - CA practically impossible distributed setup mein

üè¢ Real-World System Decisions

üí≥ Stripe (Payments)

üì± Instagram Feed

üõí Amazon Cart

üìä Stock Trading

Database Replication - Data Ki Copies

üìö Replication Kya Hai?

Replication ka matlab hai same data ki multiple copies maintain karna different machines pe. Ye ensure karta hai:

  • High Availability: Ek server down ho toh doosra le lega
  • Fault Tolerance: Data loss se bachao
  • Better Performance: Reads distribute ho sakte hain
  • Geographic Distribution: Users ke paas ka server use ho

üìä Replication Models

1Ô∏è‚É£ Single-Leader (Primary-Replica)

Kaise kaam karta hai:

  • Ek Leader/Primary - saare writes yahan
  • Multiple Replicas - reads serve karte hain
  • Leader changes log karta hai ‚Üí Replicas ko propagate hota hai

Drawback: Replication lag - replicas stale data serve kar sakte hain

2Ô∏è‚É£ Multi-Leader (Multi-Master)

Kaise kaam karta hai:

  • Multiple nodes pe writes allowed
  • Changes propagate hote hain other leaders ko
  • Multi-datacenter deployments ke liye best - local writes fast!

‚ö†Ô∏è Challenge: Conflict resolution jab same data different nodes pe modify ho!

3Ô∏è‚É£ Leaderless Replication

Kaise kaam karta hai:

  • Koi dedicated leader nahi - any node pe write ho sakta hai
  • Quorum-based reads/writes
  • DynamoDB, Cassandra use karte hain

‚è±Ô∏è Sync vs Async Replication

Synchronous Asynchronous
Write Complete Jab tak replica acknowledge na kare Local write hote hi
Data Loss Zero data loss Recent data lose ho sakta hai
Performance Slower (latency) Faster

üìù Replication Techniques

  • Statement-based: SQL statements log karke bhejte hain replicas ko
  • WAL (Write-Ahead Log) Shipping: Low-level storage engine log replicate karo
  • Logical (Row-based): Row-level changes describe karo - more flexible
üë• Multi-Master Replication Deep Dive

‚úÖ Advantages

  • High Availability: Ek master fail ho toh doosre continue karte hain
  • Distributed Access: Different locations pe masters - local writes fast
  • No Single Point of Failure: System resilient hai

‚ùå Disadvantages

  • Consistency Issues: Mostly loosely consistent (eventual consistency)
  • Performance Overhead: Eager replication complex hai, latency increase
  • Conflict Resolution: Same row different masters pe modify - kiska win?

‚öîÔ∏è Conflict Resolution Strategies

Jab same data multiple masters pe simultaneously modify ho:

üìç Last-Write-Wins (LWW)

Works for additive operations

Track causality - detect and resolve conflicts

Dynamo-style vector clocks

Caching Strategies - Speed Ka Secret

üöÄ Caching Kyun Important Hai?

"Speed is the name of the game!"

Caching se response times kam hote hain, database load kam hota hai, aur costs save hote hain!

Caching strategy depend karti hai data access patterns pe:

  • System write-heavy hai aur reads kam? (logs)
  • Data ek baar write hota hai aur baar baar read? (user profiles)
  • Data hamesha unique hai? (search queries)
üìñ Read Strategies

1Ô∏è‚É£ Cache-Aside (Lazy Loading)

üîÑ Flow:

  1. Application cache check karta hai
  2. Cache HIT: Data milta hai ‚Üí return!
  3. Cache MISS: Database se fetch ‚Üí cache mein store ‚Üí return

‚úÖ Pros:

  • Most commonly used pattern
  • Read-heavy workloads ke liye perfect
  • Cache fail ho toh bhi system chalta hai (resilient)
  • Cache data model different ho sakta hai database se

‚ùå Cons:

  • Cache aur database inconsistent ho sakti hai
  • TTL (Time-to-Live) set karna padta hai

2Ô∏è‚É£ Read-Through

üîÑ Flow:

  • Cache inline hoti hai database ke saath
  • Cache miss pe cache khud database se fetch karti hai
  • Application ko pata nahi database directly access ho rahi hai ya nahi

Cache-Aside se difference: Cache-aside mein app responsible hai fetching ke liye, read-through mein cache library/provider handle karti hai.

‚úçÔ∏è Write Strategies

1Ô∏è‚É£ Write-Through

üîÑ Flow:

  1. Data pehle cache mein write hota hai
  2. Cache synchronously database mein write karti hai
  3. Dono update hone ke baad write complete

‚úÖ Benefit: Cache aur database hamesha consistent!

‚ùå Drawback: Write latency increase (2 writes)

Example: AWS DynamoDB DAX (DynamoDB Accelerator)

2Ô∏è‚É£ Write-Around

üîÑ Flow:

  • Data directly database mein write
  • Cache bypass hoti hai write pe
  • Sirf read hone pe cache populate hoti hai

‚úÖ Best for: Data jo ek baar write ho aur rarely read ho (logs, chat messages)

3Ô∏è‚É£ Write-Back (Write-Behind)

üîÑ Flow:

  1. Data cache mein write ‚Üí immediate acknowledgment
  2. Cache asynchronously baad mein database update karti hai
  3. Batching/coalescing support ho sakta hai

‚úÖ Pros:

  • Write-heavy workloads ke liye best!
  • Super fast writes (sirf cache update)
  • Database load kam
  • Database temporarily down ho toh bhi tolerate kar sakta hai

‚ö†Ô∏è Risk: Cache fail ho toh data permanently lost!

Example: Most database storage engines (InnoDB) internally write-back use karte hain - writes pehle memory mein, phir disk pe flush.

üéØ Strategy Selection Cheat Sheet

Workload Best Strategy
Read-heavy, same data baar baar Cache-Aside + Read-Through
Strong consistency chahiye Read-Through + Write-Through
Write once, read rarely Write-Around + Read-Through
Write-heavy, performance priority Write-Back
Mixed workload Read-Through + Write-Back
üóëÔ∏è Top 8 Cache Eviction Strategies

üß† "Jab cache full ho jaye, kya delete karein?"

Eviction strategy decide karti hai ki kaun sa data cache se remove hoga!

üîπ LRU (Least Recently Used)

Jo data sabse purana accessed hai, use remove karo!

Logic: Recently accessed data firse access hone ki probability zyada hai

‚úÖ Most commonly used strategy!

üîπ MRU (Most Recently Used)

Jo data abhi access hua, use remove karo!

Use case: Jab recently accessed data dobara access nahi hota (scanning patterns)

üîπ LFU (Least Frequently Used)

Jo data sabse kam baar access hua, use remove karo!

Tracks: Access frequency (count) of each item

üîπ FIFO (First In First Out)

Jo pehle aaya, wo pehle jaayega!

Simple: Queue jaisa behaviour - oldest item removed

üìå Access patterns track nahi karta

üîπ SLRU (Segmented LRU)

Cache 2 segments mein divide - Probationary + Protected!

New items: Probationary ‚Üí Access again ‚Üí Protected

üõ°Ô∏è Hot items ko protect karta hai!

üîπ TTL (Time-to-Live)

Har item ki expiry time set karo!

After TTL: Item automatically expire ho jata hai

‚è∞ Session data, tokens ke liye best!

üîπ Two-Tiered Caching

L1: In-memory (fast) + L2: Distributed cache

Example: Local cache ‚Üí Redis ‚Üí Database

üöÄ Best of both worlds!

üîπ RR (Random Replacement)

Randomly kisi bhi item ko remove kar do!

Pros: Simple, no tracking overhead

‚ö†Ô∏è Cache Problems & Solutions

üö® Caching powerful hai but problems bhi laata hai!

Ye 4 major cache problems har system designer ko pata hone chahiye!

‚ö° 1. Cache Avalanche (Thunder Herd)

Problem: Bahut saare cache items ek saath expire ho jaate hain!

Result: Suddenly saare requests database pe ‚Üí Database CRASH! üí•

‚úÖ Solutions:

  • Different expiry times set karo (randomize TTL)
  • TTL mein random offset add karo: TTL + random(0, 300)
  • Cache warming before peak hours
üîç 2. Cache Penetration

Problem: Non-existent keys ke liye requests aati hain jo cache mein kabhi nahi milti!

Example: User search kare user_id = -1 (invalid) ‚Üí Cache miss ‚Üí DB check ‚Üí Not found ‚Üí Repeat!

‚úÖ Solutions:

  • Bloom Filter: Check if key exists before querying
  • Cache NULL: Non-existent keys ke liye bhi cache mein empty/null store karo (short TTL)
  • Input validation (reject invalid requests early)
üî• 3. Cache Breakdown (Hotspot)

Problem: Ek hot key (popular item) ki cache expire ho jati hai!

Result: Thousands of concurrent requests same key ke liye DB pe ‚Üí Overload!

‚úÖ Solutions:

  • Mutex Lock: Sirf 1 request DB se fetch kare, baaki wait karein
  • Never expire: Hot keys ko infinite TTL (manually update)
  • Logical expiry: Cache mein expiry time store karo, background refresh
üí• 4. Cache Crash (Cache Down)

Problem: Poora cache server down ho jaata hai!

Result: 100% requests directly database pe ‚Üí Instant overload!

‚úÖ Solutions:

  • Cache Cluster: Multiple cache nodes (Redis Cluster)
  • Local fallback cache: In-memory backup
  • Circuit Breaker: Limit requests when cache down
  • Rate limiting: Throttle requests to DB
üéØ Interview Questions - Caching Deep Dive

Q: "Cache Invalidation kaise handle karoge? Ye 'hardest problem in CS' kyun hai?"

Answer: Cache invalidation isliye hard hai kyunki source of truth (database) aur cache sync mein rakhna complex hai:

  • TTL-based: Simple but stale data problem - user ko old data dikh sakta hai
  • Event-driven: Database change pe cache invalidate karo (CDC - Change Data Capture)
  • Write-Through: Write pe dono update - but distributed systems mein race conditions!
  • Versioning: Har cache entry ke saath version store karo

Real Example: Facebook TAO - write-through + eventual consistency + lease-based invalidation use karta hai

Q: "E-commerce site pe flash sale - caching strategy kya hogi?"

Answer: Flash sale extremely high traffic + inventory accuracy chahiye:

  • Product Details: Read-Through + Long TTL (data rarely changes during sale)
  • Inventory Count: Write-Behind + In-memory counter (Redis INCR/DECR atomic)
  • Pre-warming: Sale start se pehle popular products cache mein load karo

Q: "API Gateway mein rate limiting kaise implement karoge?"

Q: "Service Mesh vs API Gateway - difference kya hai?"

Big companies dono use karte hain: API Gateway for external, Service Mesh for internal
